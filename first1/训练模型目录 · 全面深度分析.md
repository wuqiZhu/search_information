# 训练模型部署规划 · 全面深度分析 v2

> **战略定位变更：** v1 的核心是"训练一个模型替代 MiMo/DeepSeek"。v2 的核心是**不替代任何 API，而是做 API 做不了的事。**
>
> 你的模型不是武器，是兵种——不追求单兵作战能力，追求无处不在的覆盖。
>
> **核心指标：** 不是模型准确率，而是系统级别的「覆盖 × 延迟 × 成本」三角优化。

---

## 一、战略重新定位：从"替代"到"倍增"

### 1.1 你之前为什么方向错了

之前的规划（v1）核心假设是：**训练一个模型 → 逐渐替代 API → 省钱 + 摆脱依赖。**

这个假设有三个致命问题：

| 问题 | 具体 | 影响 |
|------|------|------|
| **质量天花板** | 1.5B 模型用 MiMo 合成数据训练，再强也追不上 MiMo/DeepSeek | 替代 = 降级 |
| **部署瓶颈** | 新加坡服务器 4GB 无 GPU，根本跑不动 7B 模型 | 训练了也部署不了 |
| **算力不匹配** | AutoDL 按小时计费，不能 24×7 跑推理 | 训练完的模型没地方住 |

### 1.2 v2 的正确方向

**你的模型不应该试图变聪明。它应该无处不在。**

它的三个不对称优势，恰好是 API 的死穴：

| 特征 | 你的小模型（1.5B 甚至更小） | MiMo/DeepSeek API |
|------|------------------------------|-------------------|
| 推理成本 | ≈ 0（CPU 都能跑） | 按 token 付费 |
| 延迟 | 50-300ms（本地） | 500ms-5s（网络） |
| 并发 | 无限副本 | 限速限频 |
| 嵌入性 | 可嵌入 pipeline 每个环节 | 只有一个 endpoint |
| 24×7 | 永远在线 | 可能熔断/超时 |
| **可改造性** | 你可以改它的行为 | 你不能改 GPT |

**所以战略不是"替代 API"，而是"用模型做放大器"：**

```
之前：单一 API 做所有分析（550条/天）
之后：小模型做 10,000 条初筛 → API 只分析 500 条最高价值的
      小模型做 pipeline 5 个环节的预判
      小模型 24×7 扫描 RSS 全量流
      小模型在每台服务器上各司其职
```

**最终效果：系统处理能力提升 20 倍，API 成本不变甚至下降。**

## 三、范式转移：你的模型到底该干什么？

### 3.1 不适合做的事（API 更擅长）

| 任务 | 为什么不适合小模型 |
|------|-------------------|
| 深度新闻分析（5 板块：态势/风向/信号/洞察/策略） | 需要复杂推理，1.5B 做不到 |
| 长文本摘要生成（500-2000 字） | 生成质量远不如 DeepSeek |
| 跨平台热点关联推理 | 需要大局观，小模型视野窄 |
| 投资影响链分析 | 需要金融知识，小模型记不住 |

### 3.2 最擅长做的事（API 做不好）

| 任务 | 为什么小模型反而有优势 | 使用场景 |
|------|----------------------|----------|
| **二分类**（相关/不相关） | 1.5B 足够，CPU 1ms | 全量新闻初筛，减少 90% 无效数据 |
| **多分类**（N 个类别选 1） | 最简单 NLP 任务，500MB 模型足够 | 新闻分类、紧急度分级 |
| **打分/排序**（0-10 分） | 回归任务，小模型学得好 | 优先级排序，决定先分析哪条 |
| **相似度判断**（和已处理的比） | 不需要语义理解，向量距离就行 | 去重、关联已有分析 |
| **格式转换/关键词提取** | 模式匹配，规则 + 小模型混合 | 从新闻中提取实体、数字、人名 |
| **高频重复劳动** | 不花钱不限速 | 24×7 监控、海量数据扫描 |

### 3.3 核心部署原则

```
API 做"质"的事：500 条精选新闻 → 深度分析
小模型做"量"的事：10,000 条原始数据 → 筛选/排序/分类

不要用模型替代 API 的分析能力。
要用模型放大 API 的覆盖范围。
```

### 阶段 0：快筛哨兵（第 1-2 周）

**一句话描述：** 训练一个极小模型，直接部署在新加坡服务器上做全量新闻初筛，把 API 从 550 条/天的分析中解放出来，只分析高价值的 500 条。

**核心逻辑：** 你现在每天只采集 11 个平台的热榜前 50，因为 API 调用要钱。有了本地免费模型，你可以采集所有内容——热榜、RSS 全量流、评论区数据——全部让模型先过一遍，筛出值得看的再送 API。

#### 4.0.1 选型决策

| 模型 | 大小 | CPU 推理速度 | 训练时间 | 推荐度 |
|------|------|-------------|----------|--------|
| **fastText** | ~10MB | 100,000 条/秒 | 10 分钟 | ⭐ 快速验证用 |
| **DistilBERT + ONNX** | ~70MB | 5,000 条/秒 | 2 小时 | ⭐⭐⭐ 主力推荐 |
| **MiniLM-L6 + ONNX** | ~22MB | 8,000 条/秒 | 1.5 小时 | ⭐⭐ 轻量备选 |
| Qwen2.5-1.5B + GPTQ | ~1.5GB | 500 条/秒 | 2-3 天 | ❌ 杀鸡用牛刀 |

**推荐：DistilBERT（教师 = 你现有的合成数据）→ ONNX 导出 → 部署到新加坡服务器。**

原因：
- 你有 160 万条标注好的分类/打分数据 → DistilBERT 够吃
- ONNX 优化后在新加坡 2 核 CPU 上每秒处理 5000 条 → 日处理 10 万条只需 20 秒
- 训练只需 2 小时，在 AutoDL 上跑一次就够

### 阶段 1：真实数据蒸馏（第 2-4 周，与阶段 0 部分并行）

**一句话描述：** 在 TrendRadar 的 AI 分析代码中加入一条钩子，每次调用 MiMo/DeepSeek 分析新闻时，自动保存（新闻输入 → AI 输出）对。每周用这些真实数据增量微调 Qwen2.5-1.5B，让它越来越懂"你关注的这些股票"。

**和阶段 0 的关系：** 阶段 0 是"做 API 做不了的事"，阶段 1 是"让 API 帮你免费训练"。两者可以同时进行。

**小偷模式的核心逻辑：**

```
正常流程：采集 → 送 API 分析 → 用结果
小偷模式：采集 → 送 API 分析 → 用结果 + 把(输入, 输出)存下来 → 周末训练
                                                                    ↓
                                   下周一模型就会了 — 类似新闻不再需要 API
```

### 阶段 2：多 LoRA 嵌入 Pipeline（第 3-8 周）

**一句话描述：** 同一个 Qwen2.5-1.5B 基座模型，加载不同的 LoRA adapter，在 pipeline 的 5 个不同环节各司其职。

**核心逻辑：** LLaMAFactory LoRA adapter 切换成本几乎为零（每个 adapter 几十 MB）。你不需要 5 个模型——你只需要 1 个基座 + 5 个 adapter。

​```
                    ┌─ LoRA A: 新闻相关性打分（这条值不值得看？）
                    ├─ LoRA B: 情绪分类（正面/负面/中性）
Qwen2.5-1.5B  ──────├─ LoRA C: 紧急度判断（现在推/晚点推/不推）
                    ├─ LoRA D: 跨平台去重（这条和昨天那条是同一事件吗？）
                    └─ LoRA E: 决策后归因（这笔买卖做错是因为模型还是黑天鹅？）
​```

#### 4.2.2 LoRA 切换机制

在新加坡服务器上部署一个轻量 LoRA 管理器：

| 项目 | 内容 |
|------|------|
| **管理脚本** | `search_information/TrendRadar/scripts/lora_manager.py` |
| **功能** | 加载基座模型 + 按任务切换 LoRA adapter |
| **内存占用** | 基座 ~3GB（4-bit 量化） + adapter ~50MB × 5 = ~3.25GB（略超新加坡 4GB） |
| **解决方案** | 一次只加载 2 个 adapter（A+D 或 B+D），其他常驻阿里云推理节点 |
| **阿里云部署** | LoRA C/E 部署在阿里云 7.1GB 服务器上作为远程推理节点 |

### 阶段 3：多节点分布式（第 6-12 周，可选）

**一句话描述：** 将你的模型部署到所有可用的服务器节点上，每个节点负责不同的数据分析任务。

**前提条件：** 阶段 2 至少 3 个 LoRA adapter 跑通且验证有效。

#### 4.3.1 节点分配方案

| 节点 | 硬件 | 运行的模型 | 负责任务 |
|------|------|-----------|----------|
| 新加坡（主） | 4GB/2 核 | DistilBERT ONNX + LoRA A+D | 全量数据初筛 + 去重 |
| 阿里云 | 7.1GB | Qwen1.5B + LoRA C+E | 紧急度判断 + 决策归因 |
| 阿里云（v2.2-fast 停产后） | 7.1GB 全释放 | Qwen1.5B + LoRA B | 批量情绪分析 |
| AutoDL（闲置时段） | RTX 4090 | Qwen2.5-7B（完整版） | 批量回测数据重分析 |
| 本地电脑（空闲时） | 视规格而定 | MiniLM ONNX | 盘后复盘、历史数据重标注 |

#### 4.3.2 通信机制

```
新加坡（主调度器） → HTTP 调用 → 阿里云推理节点
                         ← 返回结果
                         通过 notification_center 统一调度
```

**不引入消息队列**（Kafka/RabbitMQ）——你的节点就 2-3 个，HTTP 直连最简单。

## 七、前提条件与风险

### 7.1 各阶段前提条件

| 阶段 | 前提条件 | 不满足的替代方案 |
|------|---------|-----------------|
| **阶段 0** | AutoDL 可开通（用于训练）、新加坡服务器 pip 可装 onnxruntime | onnxruntime 装不上 → 用 scikit-learn 的 LogisticRegression（效果差 5% 但能用） |
| **阶段 1** | TrendRadar pipeline 正常运行、有文件写入权限 | 权限不足 → 缓冲池改写到已挂载 volume |
| **阶段 2** | Qwen2.5-1.5B 基座模型可下载 | 无法下载 → 使用 Qwen2.5-0.5B（更小但效果差 10%） |
| **阶段 3** | 阿里云服务器上 v2.2-fast 已停机释放资源 | 未停机 → 只能用新加坡服务器跑全部（4GB 内存需严格控制） |

### 7.2 风险评估

| 风险 | 概率 | 影响 | 应对方案 |
|------|------|------|----------|
| MiMo 密钥到期，无 API 可用 | 🔴 高 | 🔴 极高 | 在阶段 0 跑通前不要依赖 API——先用规则引擎做 fallback |
| ONNX 在新加坡服务器上跑不了 | 🟡 中 | 🟡 中 | 退回到 scikit-learn 逻辑回归（周期 + 2 天） |
| Qwen1.5B 基座下载被墙 | 🟡 中 | 🔴 高 | 在 AutoDL 上先下好（AutoDL 有国内镜像），转 tar 传到新加坡 |
| 阿里云 7.1GB 跑不动 Qwen1.5B + LoRA | 🟡 中 | 🟡 中 | 换 Qwen2.5-0.5B（只需 2GB），或降低精度到 8-bit |
| 增量训练导致灾难性遗忘 | 🟢 低 | 🟡 中 | 每次训练合并全部历史数据，不单独用"上周数据"训练 |
| 影子模式拖慢生产 pipeline | 🟢 低 | 🟢 低 | 影子推理用独立线程，超时 500ms 自动放弃 |

### 7.3 关键阈值仪表盘

部署后你需要持续关注的指标（不是模型指标，是系统指标）：

| 指标 | 当前值（估算） | 阶段 0 后目标 | 阶段 2 后目标 |
|------|---------------|--------------|--------------|
| 每日处理原始新闻数 | ~550 | ~10,000 | ~20,000+ |
| 每日 API 分析数 | ~550 | ~500 | ~300 |
| API 调用节省比例 | 0% | ~50% | ~70% |
| 从采集到通知的延迟 | ~30 分钟 | ~15 分钟 | ~5 分钟 |
| 漏过的重要新闻 | 未知（因为没有对照） | < 5% | < 2% |
| 新加坡服务器内存余量 | ~500MB | ~300MB | ~200MB（LoRA A+D 常驻） |

## 附录：和 v1 规划的核心区别

| 维度 | v1（旧规划） | v2（本规划） |
|------|-------------|-------------|
| **目标** | 训练模型替代 MiMo/DeepSeek | 用模型放大 API 的覆盖能力 |
| **主力模型** | Qwen2.5-7B LoRA | DistilBERT + Qwen2.5-1.5B LoRA |
| **训练数据** | 依赖阿里云合成数据 | 合成数据启动 + 真实数据持续迭代 |
| **部署位置** | 未明确（原计划 AutoDL 推理） | 新加坡 4GB CPU + 阿里云 7GB |
| **核心指标** | 模型准确率 > 90% | 系统覆盖 20 倍 + API 成本降 50-70% |
| **API 关系** | 替代 API | 给 API 当哨兵 |
| **数据流** | 一次性训练 → 部署 | 持续蒸馏（每天偷数据，每周训练） |
| **风险重心** | 模型训不出来 | MiMo 到期前跑不通初筛 |
| **最快见效** | 2 个月后（等模型训完） | 2 周后（ONNX 分类器上线） |

## 一句话总结

> **v1 想造一个更聪明的工具。v2 想打造一支无处不在的军队。**
>
> 如果你只有 2 周，做阶段 0（全量初筛）。它投入最小、见效最快、和现有系统完全不冲突。
>
> 如果你有 8 周，做阶段 0 → 1 → 2。核心是**数据不浪费、模型不闲置、API 每一分钱都花在高价值分析上**。
>
> 阶段 3（多节点）是可选项——等到你的系统覆盖到每天 50,000+ 条原始数据，才真正需要它。
