## 一、search_information (TrendRadar)

### 问题1：获取 0 个新闻 / 热榜数据为空

**原因**：
- 热榜平台 API 接口变化或限流
- NewsNow 中转服务不可用
- 网络问题导致请求超时

**处理方法**：
```powershell
# 1. 检查 NewsNow API 是否可用
python -c "import requests; r=requests.get('https://newsnow.busiyi.world/api/s?platform=weibo'); print(r.status_code, len(r.json().get('data',[])))"

# 2. 检查日志中的错误信息
Get-Content search_information\TrendRadar\logs\trendradar.log -Tail 30 | findstr -i "error fail timeout"

# 3. 如果是代理问题，检查 config.yaml 中的 proxy 配置
```

### 问题2：RSS 源抓取失败

**原因**：
- RSS 源地址变更或下线
- 网络防火墙阻止访问
- RSS 格式解析错误

**处理方法**：
```powershell
# 1. 测试单个 RSS 源连通性
python -c "import feedparser; f=feedparser.parse('https://www.phoronix.com/rss.php'); print(f'entries={len(f.entries)}, bozo={f.bozo}')"

# 2. 查看 RSS 抓取日志
Get-Content search_information\TrendRadar\logs\trendradar.log -Tail 50 | findstr -i "rss feed"

# 3. 禁用失效源：编辑 config/config.yaml，将对应源设为 enabled: false
```

### 问题3：钉钉收不到通知

**原因**：
- 钉钉机器人安全模式不匹配（关键词/加签/IP白名单）
- Webhook URL 错误或过期
- `.env` 文件中 DINGTALK_WEBHOOK 或 DINGTALK_SECRET 未设置

**处理方法**：
```powershell
# 1. 检查环境变量是否加载
cat search_information\.env | findstr DINGTALK

# 2. 测试钉钉连接
cd search_information\TrendRadar
python -c "import hmac,hashlib,base64,urllib.parse,time,requests,os;from dotenv import load_dotenv;load_dotenv('../.env');w=os.environ['DINGTALK_WEBHOOK'];s=os.environ['DINGTALK_SECRET'];ts=str(round(time.time()*1000));ss=f'{ts}\n{s}';sg=urllib.parse.quote_plus(base64.b64encode(hmac.new(s.encode(),ss.encode(),hashlib.sha256).digest()));p={'msgtype':'markdown','markdown':{'title':'测试','text':'## 测试通知'}};r=requests.post(f'{w}&timestamp={ts}&sign={sg}',json=p,timeout=10);print(r.text)"

# 3. 如果返回 "errcode": 0 表示配置正确
# 4. 如果返回 "errcode": 310000 表示安全关键词不匹配
#    解决：在钉钉机器人设置中添加关键词，或在消息中包含该关键词
```

### 问题4：AI 分析失败

**原因**：
- DeepSeek API Key 过期或余额不足
- API 请求超时
- 模型名称配置错误

**处理方法**：
```powershell
# 1. 检查 API Key 配置
cat search_information\.env | findstr DEEPSEEK

# 2. 测试 API 连通性
python -c "import requests; r=requests.post('https://api.deepseek.com/v1/chat/completions', headers={'Authorization': 'Bearer YOUR_KEY'}, json={'model':'deepseek-chat','messages':[{'role':'user','content':'test'}]}, timeout=30); print(r.status_code, r.json().get('choices',[{}])[0].get('message',{}).get('content','')[:50])"

# 3. 如果返回 401：API Key 无效
# 4. 如果返回 429：请求频率过高，等待后重试
# 5. 如果返回 200 但内容为空：检查 model 名称是否正确
```

### 问题5：关注词匹配不到内容

**原因**：
- 关注词太长太精确
- 关注词语法错误（正则表达式格式不对）
- 相关性阈值太高

**处理方法**：
```powershell
# 1. 使用短关键词测试
# 将 "嵌入式Linux实习" 改为 "嵌入式"、"实习"

# 2. 检查关注词语法
# 正确：华为、/华为|小米/、+嵌入式、!外包
# 错误：华为小米（会被当作一个完整词匹配）

# 3. 编辑关注词配置
notepad search_information\TrendRadar\config\frequency_words.txt
```

### 问题6：emoji 编码错误导致 Obsidian 同步崩溃

**原因**：`obsidian_sync.py` 中使用了 Unicode emoji 字符，在某些终端编码下会报错。

**处理方法**：
```powershell
# 编辑文件 search_information/TrendRadar/dashboard/obsidian_sync.py
# 第211行：print(f"\u2705 同步...") → print(f"[OK] 同步...")
# 第214行：print(f"\u274c 同步失败...") → print(f"[FAIL] 同步失败...")
```

## 二、analyse_information

### 问题1：RSS 拉取返回 0 条

**原因**：
- RSS 源地址失效
- 已处理的 URL 重复（数据库去重）
- 网络问题

**处理方法**：
```powershell
# 1. 检查 RSS 源配置
cat analyse_information\analyzer\config.yaml | findstr -A2 "url:"

# 2. 测试单个 RSS 源
python -c "import feedparser; f=feedparser.parse('https://hnrss.org/frontpage'); print(f'entries={len(f.entries)}')"

# 3. 如果是去重问题，查看已处理 URL 数量
cd analyse_information
python -c "from analyzer.db import get_stats; import json; print(json.dumps(get_stats(), indent=2))"
```

### 问题2：AI 分析返回 "not_relevant"

**原因**：
- 文章内容与关键词不匹配
- 相关性阈值设置太高
- AI 模型判断保守

**处理方法**：
```powershell
# 1. 降低相关性阈值
# 编辑 analyse_information/analyzer/config.yaml
# relevance.min_score: 0.3 → 0.2

# 2. 添加更多相关关键词
# 编辑 config.yaml 中 relevance.keywords 部分

# 3. 使用反馈机制调整偏好
cd analyse_information
python -c "from analyzer.db import set_feedback; set_feedback('URL', 'useful')"
```

### 问题3：Obsidian 笔记未生成

**原因**：
- Obsidian 路径配置错误
- 目录权限问题
- 分析结果为 not_relevant（不会生成笔记）

**处理方法**：
```powershell
# 1. 检查 Obsidian 路径配置
cat analyse_information\analyzer\config.yaml | findstr obsidian

# 2. 检查目录是否存在
Test-Path analyse_information\knowledge_base\obsidian

# 3. 手动触发笔记生成
cd analyse_information
python analyzer/pipeline.py --url "https://example.com/article"
```

### 问题4：Defuddle 内容提取失败

**原因**：
- 目标网站有反爬机制
- 页面需要 JavaScript 渲染
- 网络超时

**处理方法**：
```powershell
# 查看错误日志
Get-Content analyse_information\shared\logs\scheduler.log -Tail 30 | findstr -i "defuddle extract fail"

# Defuddle 失败会自动降级到 BeautifulSoup，通常不需要手动处理
# 如果两个都失败，说明目标网站确实无法抓取
```

### 问题5：Token 费用异常高

**原因**：
- 分析的文章数量过多
- 文章内容过长（触发截断前消耗大量 token）
- 使用了昂贵的模型

**处理方法**：
```powershell
# 1. 查看 Token 统计
Get-Content analyse_information\shared\logs\scheduler.log -Tail 50 | findstr -i "token cost"

# 2. 限制分析数量（在 pipeline.py 参数中控制）
python analyzer/pipeline.py --rss --limit 10

# 3. 切换到更便宜的模型
# 编辑 analyse_information/analyzer/config.yaml
# ai.model: "deepseek-chat" (便宜) vs "deepseek-reasoner" (贵)
```

## 三、find_job

### 问题1：Boss 直聘抓取返回 0 条

**原因**：
- BOSS_COOKIE 过期或无效
- 触发了安全验证（滑块/验证码）
- 抓取模式不匹配

**处理方法**：
```powershell
# 1. 检查 Cookie 是否设置
cd find_job
python -c "from job_scraper.config import get_config; c=get_config(); print('Cookie 已设置' if c.get('boss_cookie') else 'Cookie 未设置！')"

# 2. 更新 Cookie
python scripts/get_cookie.py

# 3. 切换抓取模式
# 编辑 find_job/.env，设置 SCRAPE_MODE=http（跳过浏览器模式）
# 或 SCRAPE_MODE=browser（仅浏览器模式）

# 4. 查看日志
Get-Content find_job\data\logs\scraper.log -Tail 30 | findstr -i "cookie error 403 36"
```

### 问题2：收到"Cookie异常提醒"通知

**原因**：
- Boss 直聘检测到异常登录
- Cookie 过期需要重新登录
- 触发了安全验证

**处理方法**：
```powershell
# 1. 在浏览器中打开 zhipin.com
# 2. 登录账号，完成安全验证（滑块/短信）
# 3. 确认能正常浏览职位页面
# 4. 更新 Cookie
cd find_job
python scripts/get_cookie.py

# 5. 如果使用 GitHub Actions，同步更新 Secrets 中的 BOSS_COOKIE
```

### 问题3：AI 评分全部失败

**原因**：
- DEEPSEEK_API_KEY 未设置或无效
- API 请求超时
- curl_cffi 未安装导致 TLS 指纹被识别

**处理方法**：
```powershell
# 1. 检查 API Key
cd find_job
python -c "from job_scraper.config import get_config; c=get_config(); print(f'Key: {c[\"deepseek_api_key\"][:10]}...' if c['deepseek_api_key'] else 'Key 未设置！')"

# 2. 测试 API 连通性
python -c "from job_scraper.config import get_config; import requests; c=get_config(); r=requests.post(f'{c[\"deepseek_base_url\"]}/chat/completions', headers={'Authorization': f'Bearer {c[\"deepseek_api_key\"]}'}, json={'model':c['deepseek_model'],'messages':[{'role':'user','content':'test'}]}, timeout=30); print(r.status_code)"

# 3. 安装 curl_cffi（提升抓取成功率）
pip install curl_cffi
```

### 问题4：猎聘/拉勾/牛客抓取失败

**原因**：
- 这些平台使用 DrissionPage 浏览器自动化
- DrissionPage 未安装
- Chrome/Chromium 未安装或版本不匹配

**处理方法**：
```powershell
# 1. 安装 DrissionPage
pip install DrissionPage

# 2. 确认 Chrome 已安装
# 打开 Chrome，访问 chrome://version/ 查看版本

# 3. 查看错误日志
Get-Content find_job\data\logs\scraper.log -Tail 30 | findstr -i "drission chrome browser"

# 4. 如果 DrissionPage 持续失败，系统会自动降级到 HTTP 模式（仅 Boss 支持）
```

### 问题5：钉钉通知收不到高分岗位

**原因**：
- DINGTALK_WEBHOOK 或 DINGTALK_SECRET 未设置
- 钉钉安全关键词不匹配
- 评分阈值太高（默认 80 分）

**处理方法**：
```powershell
# 1. 检查环境变量
cat find_job\.env | findstr DINGTALK

# 2. 测试通知
cd find_job
python -c "from job_scraper.notifier import send_markdown; print(send_markdown('测试', '## 测试通知'))"

# 3. 降低评分阈值
# 编辑 find_job/job_scraper/constants.py
# SCORE_THRESHOLD = 80 → 70
```

### 问题6：GitHub Actions 工作流失败

**原因**：
- Secrets 中的环境变量未配置
- Python 依赖安装失败
- Cookie 过期

**处理方法**：
```powershell
# 1. 检查 GitHub Secrets 配置
# 需要设置：BOSS_COOKIE, DINGTALK_WEBHOOK, DINGTALK_SECRET, DEEPSEEK_API_KEY

# 2. 手动触发工作流测试
# GitHub 仓库 → Actions → Job Scraper → Run workflow

# 3. 查看工作流日志
# GitHub 仓库 → Actions → 点击失败的工作流 → 查看详细日志

# 4. 常见错误：
# "BOSS_COOKIE is not set" → Secrets 未配置
# "HTTP 403" → Cookie 过期
# "ModuleNotFoundError" → requirements.txt 缺少依赖
```

### 问题7：已见岗位太多，新岗位变少

**原因**：
- `data/seen_jobs.json` 积累了大量历史记录
- 同一个岗位被反复匹配但已标记为"已见"

**处理方法**：
```powershell
# 清理已见岗位记录（下次运行会重新抓取和评分）
Remove-Item find_job\data\seen_jobs.json -Force
cd find_job
python -m job_scraper.main
```

## 五、服务器部署（DigitalOcean Docker）

### 问题1：DeepSeek API Token 失控消耗（严重）

**原因**：
- find_job 的 `github_scraper.py` 对每个新岗位都调用一次 DeepSeek API
- 46个搜索查询 × 每个返回10-30个岗位 = 单次运行可能产生 300-500 次 API 调用
- 调用间隔只有 2-5 秒，无全局调用上限
- 5分钟内消耗 50 元 Token

**处理方法**：
```bash
# 1. 立即删除所有 DeepSeek API Key
# 在各项目的 .env 文件中清空 DEEPSEEK_API_KEY

# 2. 在 DeepSeek 平台轮换 API Key
# 访问 https://platform.deepseek.com/ 生成新 Key

# 3. 代码已修复：添加了以下保护机制
# - MAX_API_CALLS_PER_RUN=30（单次运行最多30次调用）
# - API_CALL_DELAY_MIN/MAX=5/10（调用间隔5-10秒）
# - MAX_CALLS_PER_HOUR=60（Flask服务每小时限制）
# - 各项目都有 TokenTracker 追踪消耗
```

### 问题2：TrendRadar 容器启动失败（pytz 缺失）

**原因**：
- `requirements.txt` 缺少 `pytz` 和 `litellm` 依赖
- 容器启动时 `import pytz` 失败

**处理方法**：
```bash
# 1. 服务器拉取最新代码
cd /root/projects/search_information && git pull

# 2. 重新构建镜像
cd /root/projects && docker-compose build --no-cache trendradar

# 3. 重启容器
docker-compose up -d trendradar
```

### 问题3：invest-backend 容器显示帮助信息后退出

**原因**：
- Dockerfile 的 CMD 是 `python scripts/main_cli.py --help`
- 应该运行 `python scripts/news_api_server.py`（HTTP API 服务）

**处理方法**：
```bash
# 1. 服务器拉取最新代码（Dockerfile 已修复）
cd /root/projects/invest && git pull

# 2. 重新构建并重启
cd /root/projects && docker-compose build --no-cache invest-backend
docker-compose up -d invest-backend
```

### 问题4：analyser 容器每分钟重启

**原因**：
- Pipeline 跑完后正常退出（exit code 0）
- `restart: always` 导致容器自动重启，形成循环

**处理方法**：
```bash
# 修改 docker-compose.yml，添加休眠循环
# 让 analyser 跑完 Pipeline 后休眠1小时再重启
sed -i 's|command:.*|command: sh -c "python analyzer/pipeline.py --rss \\&\\& sleep 3600"|' /root/projects/docker-compose.yml

# 重启
cd /root/projects && docker-compose up -d analyser
```

### 问题5：服务器 SSH 终端无法捕获命令输出

**原因**：
- 服务器的 `/etc/bash.bashrc` 中有损坏的 `apt_pkg` 模块
- 每次 shell 初始化都会输出错误信息，干扰终端输出捕获

**处理方法**：
```bash
# 方案A：使用 Python paramiko 脚本执行远程命令（推荐）
# 本地运行 check_server.py 脚本

# 方案B：修复服务器 apt_pkg 问题
ssh root@服务器IP "mv /usr/lib/cnf-update-db /usr/lib/cnf-update-db.bak"

# 方案C：使用非交互式 SSH
ssh -i key_file root@服务器IP "docker ps -a"
```

### 问题6：Docker 镜像未更新（git pull 后）

**原因**：
- `git pull` 只更新了代码文件
- Docker 镜像是在 `docker-compose build` 时构建的
- 需要重新构建才能应用代码修改

**处理方法**：
```bash
# 重新构建所有镜像
cd /root/projects && docker-compose build --no-cache

# 重启所有容器
docker-compose up -d

# 只重建某个服务
docker-compose build --no-cache trendradar
docker-compose up -d trendradar
```

### 问题2：端口被占用

**原因**：之前的进程未正确退出，端口仍被占用。

**处理方法**：
```powershell
# 查找占用端口的进程
netstat -ano | findstr :5678

# 杀掉指定进程
taskkill /F /PID <进程ID>

# 或杀掉所有 Python 进程
taskkill /F /IM python.exe
```

### 问题3：编码错误（中文乱码）

**原因**：Windows 终端默认编码不是 UTF-8。

**处理方法**：
```powershell
# 临时设置（当前终端有效）
chcp 65001

# 永久设置
# Windows 设置 → 时间和语言 → 语言和区域 → 管理语言设置 → 更改系统区域设置 → 勾选"Beta: 使用 Unicode UTF-8 提供全球语言支持"
```

### 问题4：磁盘空间不足

**原因**：SQLite 数据库和日志文件持续增长。

**处理方法**：

```powershell
# 查看各项目数据大小
Get-ChildItem search_information\TrendRadar\output -Recurse | Measure-Object -Property Length -Sum
Get-ChildItem find_job\data -Recurse | Measure-Object -Property Length -Sum
Get-ChildItem analyse_information\knowledge_base -Recurse | Measure-Object -Property Length -Sum

# 清理旧数据（保留最近 30 天）
# TrendRadar 支持在 config.yaml 中设置 data_retention_days
```

## 六、关键词配置问题

### 问题1：关键词匹配不到内容

**原因**：
- 关键词太长太精确（如 "嵌入式Linux实习" 会被当作一个完整词匹配）
- 关键词语法错误（正则表达式格式不对）
- 热榜新闻标题不含该关键词

**处理方法**：
```powershell
# 1. 使用短关键词测试
# 将 "嵌入式Linux实习" 改为 "嵌入式"、"实习" 两个独立关键词

# 2. 检查关注词语法
# 正确：华为、/华为|小米/、+嵌入式、!外包
# 错误：华为小米（会被当作一个完整词匹配）

# 3. 编辑关注词配置
notepad search_information\TrendRadar\config\frequency_words.txt

# 4. 重启 TrendRadar 使配置生效
```

### 问题2：MCP 深度搜索无结果

**原因**：
- 没有可用的新闻数据（需要先运行爬虫生成数据）
- 搜索范围太窄（只搜了今天，可以扩大到 last_week）
- 关键词不匹配

**处理方法**：
```powershell
# 1. 检查是否有可用数据
# 通过 MCP 工具：list_available_dates(source="local")

# 2. 扩大搜索范围
# 将 date_range 从 "today" 改为 "last_week" 或 "last_month"

# 3. 使用模糊搜索
# 将 search_mode 从 "keyword" 改为 "fuzzy"

# 4. 同时搜索 RSS
# 添加 include_rss=True 参数
```

### 问题3：第二层关键词推送太多（人事变动/融资等）

**原因**：
- "裁员"、"融资" 等词太泛，会匹配到大量无关内容
- 需要结合具体公司名使用

**处理方法**：
```
# 方案1：使用组合关键词（在频率词中用空格分隔）
# 改为：华为 裁员、特斯拉 融资、字节 IPO

# 方案2：使用 MCP 按需搜索（推荐）
# 不放在频率词中，而是需要时通过 MCP 搜索
# search_news(query="华为 裁员", search_mode="keyword")
```
