================================================================
一、项目概述
================================================================

1. 项目名称
   信息分析与知识沉淀流水线 (Information Analysis & Knowledge Pipeline)

2. 项目目标
   - 从海量信息中筛选与嵌入式Linux/BSP/驱动开发相关的内容
   - 将技术信息翻译成大白话，降低理解门槛
   - 验证信息的真实性和可信度
   - 将优质内容沉淀到 Obsidian 知识库
   - 通过反馈机制让系统越用越聪明

3. 与上游项目的关系
   上游项目：信息获取（TrendRadar）→ 自动化（n8n）→ 通知（钉钉/邮箱）
   本项目：信息分析 → 质量检测 → 知识沉淀

   数据流向：
   TrendRadar 检测信号 → n8n 工作流 → 本项目分析 → 输出结果

四、核心模块详解
================================================================

4.1 pipeline.py — 主流程入口
─────────────────────────────
职责：命令行解析、流程编排、并发控制、报告生成

核心函数：
- load_config()           加载 config.yaml 配置
- setup_logging()         配置日志（RotatingFileHandler）
- process_single_url()    处理单个 URL 的完整流程
- process_batch()         并发批量处理多个 URL
- process_rss()           从 RSS 订阅源拉取并处理
- process_from_json()     从 JSON 文件读取 URL 列表
- process_from_n8n_webhook()  处理 n8n Webhook 数据
- print_report()          打印处理报告（含 Token 统计）

并发机制：
- 使用 ThreadPoolExecutor，默认 3 个线程
- 可通过 --workers 参数调整
- 所有线程共享同一个 AIAnalyzer 实例（TokenTracker 有线程锁）


4.2 ai_analyzer.py — AI 分析模块
─────────────────────────────────
职责：调用 DeepSeek API 完成相关性评估、大白话翻译、智能分类

核心类：
- TokenTracker       Token 用量统计（线程安全）
- AIAnalyzer         AI 分析器

分析流程（统一模式）：
1. 构建反馈上下文（从数据库加载用户标记的正/负样本）
2. 智能截断内容（保留开头+结尾，中间标注省略）
3. 构建统一 Prompt（相关性+翻译+分类一次完成）
4. 调用 DeepSeek API（1 次调用完成 3 个任务）
5. 解析 JSON 响应
6. 如果翻译或分类为空，自动补充

降级策略：
- 统一 Prompt 失败 → 分步 Prompt（先判断相关性，再翻译+分类）
- API 不可用 → 关键词匹配
- 全部失败 → 返回错误

相关性评分标准：
- 10分：直接相关（嵌入式Linux驱动开发、BSP移植、设备树配置等）
- 7-9分：高度相关（Linux内核、U-Boot、交叉编译工具链等）
- 4-6分：间接相关（通用Linux知识、编程技巧、硬件基础等）
- 1-3分：基本不相关（前端开发、Web后端等）
- 0分：完全不相关
- >= 6 分才进入翻译和沉淀流程


4.3 defuddle/extractor.py — 内容提取模块
─────────────────────────────────────────
职责：从 URL 提取干净的正文内容

四级降级策略：
1. Defuddle（npm 工具，最干净）→ 如果失败 ↓
2. 正文区域识别（article/main/content 标签）→ 如果内容太短 ↓
3. BeautifulSoup 全文提取 → 如果还是不行 ↓
4. 纯文本提取

正文区域识别逻辑：
- 按优先级尝试：article > main > [role='main'] > .post-content 等
- 找到第一个文本长度 > 200 的区域
- 删除 nav/footer/header/aside/form 等非正文标签
- 保留 h1-h5/p/li/pre/code/blockquote/table 等结构化标签
- 输出 Markdown 格式


4.4 db.py — 数据库模块
───────────────────────
职责：SQLite 数据库操作，包括 URL 去重、全文搜索、反馈记录

数据库表：
- processed_urls     主表，存储所有处理记录
- articles_fts       FTS5 全文搜索虚拟表

关键特性：
- WAL 模式（读写不阻塞）
- 全局线程锁（防止并发写入冲突）
- FTS5 全文索引（标题、理由、翻译内容）
- 自动触发器（INSERT/UPDATE/DELETE 自动同步 FTS 索引）

核心函数：
- init_db()              初始化数据库和 FTS 索引
- is_url_processed()     检查 URL 是否已处理（去重）
- save_result()          保存处理结果
- search_articles()      全文搜索（FTS5 → LIKE 降级）
- set_feedback()         设置反馈标记
- get_user_preferences() 获取用户偏好（正/负样本）
- get_stats()            获取统计数据


4.5 knowledge_builder.py — 知识沉淀模块
────────────────────────────────────────
职责：生成 Obsidian 笔记、保存分析结果

笔记格式（含 YAML frontmatter）：
- title:     文章标题
- source:    原文链接
- date:      采集日期
- category:  分类
- score:     相关性评分
- tags:      标签列表

笔记内容：
- 一句话总结
- 大白话翻译
- 原始信息（来源、时间、评分、分类）
- 关键要点
- 相关笔记（Obsidian 双向链接 [[笔记名]]）

文件命名规则：
- 格式：{时间戳}_{标题}.md
- 时间戳精确到秒，防止同名覆盖
- 同名文件自动加序号


4.6 rss_fetcher.py — RSS 订阅模块
───────────────────────────────────
职责：从配置的 RSS 源拉取文章列表

支持格式：
- Atom (xmlns:atom)
- RSS 2.0 (item)

特性：
- 自动去重（基于 URL hash）
- 支持多个 RSS 源
- 在 config.yaml 中配置


4.7 digest.py — 摘要生成模块
─────────────────────────────
职责：生成每日/每周精选摘要

每日摘要内容：
- 今日处理了多少篇文章
- 精选文章列表（标题、评分、分类、原文链接、笔记链接）
- 分类统计

每周摘要内容：
- 本周 Top 10 精选
- 分类分布（带柱状图）
- 统计数据（平均评分、最高评分）

七、使用命令大全
================================================================

所有命令在项目根目录执行：
cd analyse_information

7.1 核心处理命令

# 处理单个 URL
python analyzer/pipeline.py --url "https://example.com/article"

# 处理多个 URL（并发）
python analyzer/pipeline.py --urls "https://url1.com" "https://url2.com" "https://url3.com"

# 从 JSON 文件批量处理
python analyzer/pipeline.py --json urls.json

# 指定并发线程数（默认 3）
python analyzer/pipeline.py --urls "url1" "url2" "url3" --workers 5

# 从 RSS 订阅源自动拉取并处理
python analyzer/pipeline.py --rss

7.2 搜索与反馈命令

# 全文搜索已沉淀的文章
python analyzer/pipeline.py --search "GPIO"
python analyzer/pipeline.py --search "设备树"
python analyzer/pipeline.py --search "RISC-V"

# 标记文章为有用
python analyzer/pipeline.py --feedback "https://example.com/article" useful

# 标记文章为没用
python analyzer/pipeline.py --feedback "https://example.com/article" not_useful

# 查看用户偏好统计
python analyzer/pipeline.py --preferences

7.3 摘要命令

# 生成每日摘要
python analyzer/pipeline.py --digest

# 生成每周摘要
python analyzer/pipeline.py --weekly-digest

7.4 统计命令

# 查看历史统计
python analyzer/pipeline.py --stats

7.5 n8n 集成命令

# 处理 n8n Webhook 数据
python analyzer/pipeline.py --webhook-data '{"url":"https://...","title":"...","content":"..."}'

7.6 辅助参数

# 指定配置文件
python analyzer/pipeline.py --config /path/to/config.yaml --url "..."

# 查看帮助
python analyzer/pipeline.py --help

九、数据流详解
================================================================

9.1 单个 URL 处理流程

URL 输入
  ↓
[1/4] 内容提取
  - 尝试 Defuddle → 正文区域识别 → BeautifulSoup → 纯文本
  - 保存原始内容到 knowledge_base/raw/
  ↓
[2/4] AI 分析（1 次 API 调用完成 3 个任务）
  - 加载用户偏好（正/负样本）
  - 智能截断内容（保留开头+结尾）
  - 构建统一 Prompt
  - 调用 DeepSeek API
  - 解析结果：相关性评分 + 大白话翻译 + 分类
  ↓
  评分 >= 6？
  ├─ 否 → 标记为 not_relevant，保存到数据库，结束
  └─ 是 ↓
[3/4] 保存分析结果
  - 保存分析结果 JSON 到 knowledge_base/analyzed/
  - 保存翻译结果到 knowledge_base/translated/
  ↓
[3.5/4] 搜索相关笔记
  - 在数据库中搜索标题相似的已有笔记
  - 最多返回 5 篇相关笔记
  ↓
[4/4] 沉淀到 Obsidian 知识库
  - 生成带 frontmatter 的 Markdown 笔记
  - 包含：一句话总结、大白话翻译、关键要点、相关笔记链接
  - 按分类归档到对应目录
  - 保存到数据库
  ↓
处理完成，打印报告（含 Token 统计）

9.2 RSS 处理流程

RSS 订阅源配置 (config.yaml)
  ↓
拉取所有 RSS 源的最新文章
  ↓
去重（基于 URL hash）
  ↓
提取所有 URL
  ↓
进入批量处理流程（同上）

9.3 反馈自学习流程

用户标记文章 useful/not_useful
  ↓
保存到数据库 feedback 字段
  ↓
下次处理新文章时
  ↓
从数据库加载用户偏好
  ↓
构建反馈上下文：
  "用户喜欢的文章：[标题1, 标题2, ...]"
  "用户不喜欢的文章：[标题3, ...]"
  ↓
注入到 AI Prompt 中
  ↓
AI 根据用户偏好调整判断标准

十一、数据库结构
================================================================

表名：processed_urls
┌──────────────────┬─────────┬────────────────────────┐
│ 字段              │ 类型    │ 说明                    │
├──────────────────┼─────────┼────────────────────────┤
│ id               │ INTEGER │ 主键，自增              │
│ url              │ TEXT    │ 原始 URL                │
│ url_hash         │ TEXT    │ URL 的 MD5 哈希（唯一）  │
│ title            │ TEXT    │ 文章标题                │
│ status           │ TEXT    │ 状态                    │
│ score            │ REAL    │ 相关性评分 (0-10)       │
│ category         │ TEXT    │ 分类                    │
│ reason           │ TEXT    │ AI 给出的理由            │
│ obsidian_path    │ TEXT    │ Obsidian 笔记路径       │
│ feedback         │ TEXT    │ 用户反馈 (useful/not_useful) │
│ translation_cache│ TEXT    │ 翻译缓存（用于全文搜索） │
│ created_at       │ TEXT    │ 创建时间                │
│ updated_at       │ TEXT    │ 更新时间                │
└──────────────────┴─────────┴────────────────────────┘

状态值：
- success        成功沉淀
- not_relevant   不相关
- extract_failed 内容提取失败
- ai_failed      AI 分析失败
- error          其他错误
- skipped        已跳过（重复 URL）

全文搜索：
- 使用 SQLite FTS5 引擎
- 索引字段：title, reason, translation_cache
- 自动触发器同步索引


================================================================
十二、Token 用量与费用
================================================================

每次 API 调用会记录 Token 用量：
- 输入 Token (prompt_tokens)
- 输出 Token (completion_tokens)
- 总 Token (total_tokens)
- 预估费用（按 DeepSeek 定价 ¥0.002/1000 tokens）

典型处理一篇相关文章：
- 1 次 API 调用
- 输入约 1200 tokens
- 输出约 600 tokens
- 总计约 1800 tokens
- 预估费用约 ¥0.004

处理不相关内容：
- 1 次 API 调用
- 输入约 1200 tokens
- 输出约 60 tokens
- 总计约 1260 tokens
- 预估费用约 ¥0.003

批量处理 100 篇文章的预估费用：约 ¥0.3 - ¥0.4

十三、降级策略详解
================================================================

13.1 内容提取降级链

Defuddle (npm 工具，最干净)
  ↓ 失败/超时
正文区域识别 (article/main/content 标签)
  ↓ 内容太短 (< 100 字)
BeautifulSoup 全文提取
  ↓ 异常
纯文本提取

13.2 AI 分析降级链

统一 Prompt (相关性+翻译+分类，1 次调用)
  ↓ 失败
分步 Prompt (先判断相关性，再翻译+分类)
  ↓ API 不可用
关键词匹配 (基于 config.yaml 中的关键词列表)

13.3 分类降级链

AI 分类
  ↓ 失败
关键词匹配 (根据 category 中的 keywords 打分)

13.4 全文搜索降级链

FTS5 全文索引
  ↓ 异常
LIKE 模糊匹配 (title/reason/translation_cache)

十七、使用者需要注意的事项
================================================================

17.1 首次使用

1. 确保 Python 3.8+ 已安装
2. 安装依赖：pip install pyyaml requests beautifulsoup4 python-dotenv lxml
3. 创建 .env 文件，填入 DeepSeek API Key
4. 运行测试：python analyzer/pipeline.py --url "https://example.com"

17.2 API Key 安全

- API Key 存放在 .env 文件中
- .env 已在 .gitignore 中，不会被提交
- 不要将 API Key 硬编码到代码中
- 不要在公开场合分享 .env 文件

17.3 成本控制

- 每篇文章约消耗 1200-1800 tokens，费用约 ¥0.003-0.004
- 不相关的内容只消耗约 1260 tokens
- 使用 --stats 查看历史处理量
- 使用 --feedback 标记文章，让 AI 更精准，减少无效处理

17.4 去重机制

- 相同 URL 不会重复处理
- 第二次处理同一 URL 会直接跳过
- 如需重新处理，需要手动删除数据库记录

17.5 Obsidian 使用

- 生成的笔记在 knowledge_base/obsidian/ 目录下
- 用 Obsidian 打开该目录即可浏览
- 笔记中的 [[笔记名]] 是双向链接，点击可跳转
- 每日/每周摘要在 00-收件箱 目录下

17.6 RSS 订阅

- RSS 源在 config.yaml 的 rss.feeds 中配置
- 可以添加/删除/修改订阅源
- 使用 --rss 命令拉取最新文章
- 建议配合 Windows 任务计划定时执行

17.7 反馈机制

- 标记文章为 useful/not_useful
- AI 会根据你的偏好调整判断标准
- 用得越多，判断越准
- 使用 --preferences 查看偏好统计

17.8 日志查看

- 日志文件：shared/logs/analyzer.log
- 使用文本编辑器或 Get-Content 命令查看
- 日志会自动轮转，不会无限增长

17.9 数据库维护

- 数据库文件：shared/data/analyzer.db
- 使用 SQLite 工具可直接查询
- 使用 --stats 查看统计数据
- 如需重置，删除 analyzer.db 文件即可

17.10 常见问题

Q: DeepSeek API 返回 402 错误？
A: 账户余额不足，请登录 DeepSeek 控制台充值。

Q: 提取的内容质量不好？
A: 某些网站需要登录或 JS 渲染，当前无法处理。可以手动复制内容，通过 --webhook-data 传入。

Q: 处理速度很慢？
A: 主要瓶颈是 AI API 调用，每篇约 10-20 秒。可增加 --workers 数量提高并发。

Q: 如何添加新的分类？
A: 编辑 config.yaml 的 categories 部分，添加新的分类名称和关键词。

Q: 如何修改相关性判断标准？
A: 编辑 config.yaml 的 relevance.keywords，添加或删除关键词。
