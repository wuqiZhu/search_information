## 一、短期（本周，2-3 小时）

### 目标：完善数据可视化，让 Grafana 面板更专业

你已经完成了 InfluxDB 和 Grafana 的对接，传感器数据已入库。下一步：

1. **优化 Grafana 仪表盘**  
   - 创建多个 Panel，分别显示温度曲线、湿度曲线、烟雾状态、光敏状态、继电器状态。  
   - 使用 `stat` 面板显示当前最新温湿度值。  
   - 添加告警阈值线（例如温度 32℃ 红线）。  
   - 设置自动刷新（如每 5 秒）。

2. **增加烟雾告警的事件记录**  
   - 在 Python 脚本中，单独订阅 `device/alert` 主题，将告警事件写入 InfluxDB 的另一个 measurement（如 `alerts`），或作为 `sensor_data` 中的一个字段（如 `alert_smoke=1`）。  
   - 在 Grafana 中创建一个表格面板，显示最近的告警记录（时间、类型、级别）。

3. **修复光敏阈值**  
   - 实测光敏 ADC 值范围，调整 `rpc_server.c` 中的阈值（例如 `raw < 500` 为亮），让 QT 和 Grafana 显示正确的亮/暗。

## 二、中期（下周，3-4 小时）

### 目标：实现智能告警推送 + 远程联动

你已经实现了设备端自动化，但告警仍停留在 MQTT 消息。可以增加推送服务：

1. **钉钉/企业微信机器人告警**  
   - 在云服务器上编写一个 Python 脚本（或 Node-RED 流），订阅 `device/alert` 主题。  
   - 当收到告警时，调用钉钉/企业微信 Webhook API 发送消息到群。  
   - 可设置不同级别（如烟雾超标 → 紧急告警，温度过高 → 提醒）。

2. **增加手动/自动的“全关”场景**  
   - 定义一个 MQTT 指令 `{"method":"all_off"}`，让 `mqtt_bridge` 同时关闭风扇和 LED 灯。  
   - 在 QT 界面上添加“全关”按钮。

3. **优化烟雾联动逻辑**  
   - 当前烟雾报警后风扇强制开启 30 秒，可以改为：烟雾持续报警时风扇一直开，恢复正常后延迟 10 秒关闭。

## 三、长期（持续优化，1 个月）

### 目标：打造统一 Web 控制台，提升系统完整性

1. **部署 Node-RED**  
   - 通过 1Panel 一键安装 Node-RED。  
   - 创建 Dashboard：显示传感器实时数据（曲线、仪表），并放置开关按钮控制 LED、风扇、照明。  
   - 实现规则引擎可视化配置（例如：当温度 > 32℃ → 开风扇）。  
   - 优点：无需修改代码，拖拽即可扩展联动逻辑。

2. **接入 ThingsBoard 或 EMQX（可选）**  
   - ThingsBoard 是开源物联网平台，提供设备管理、规则链、仪表盘等，比自建 InfluxDB+Grafana 更完整。  
   - 但需要一定学习成本，可以作为后续升级方向。

3. **完善文档与视频**  
   - 编写完整的项目文档（硬件接线、软件配置、API 说明）。  
   - 录制演示视频，上传到 B站/YouTube，作为作品展示和面试素材。

## 四、其他建议

- **性能优化**：`mqtt_bridge` 的定时上报间隔（5 秒）可能略频繁，可改为 10 秒，减少网络负载。  
- **代码重构**：将硬编码的阈值（如 2000、30 秒）改为可配置的宏或环境变量。  
- **增加更多传感器**：如空气质量、二氧化碳等，扩展应用场景。
